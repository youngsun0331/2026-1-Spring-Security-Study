## 토큰 기반 인증이란?
### 홍대 축제 비유
색깔 팔찌를 전달하고 학생회는 팔찌 색깔만 확인한다

-> 상태를 갖고다닌다

서버에서 유저에게 토큰을 주고 사용자를 저장하지 않는다

-> 토큰 내용만 검증한다 (인가영역)
-> Stateless

### 토큰 규약
1. 형식이 정해져있어야 함.
2. 변조하기 어려워야 한다.<br>
누가 만들든 같은 규격으로 읽을 수 있고,<br>
토큰 내용을 마음대로 바꾸거나 위조 할 수 없게 만든다

-> JWT(Json Web Token)

## JWT?

1. Header - 어떤 알고리즘으로 서명되었는가?
2. Payload - 사용자 정보
3. Signature - 1/2를 비밀 키로 서명한 값

### JWT로 사용자 구분하기
0. 유저가 토큰을 보낸다
    + 보통 authorization 헤더에 담아서 보낸다
1. 토큰을 꺼낸다
2. 토큰을 파싱한다
3. 변조 여부를 확인한다
    + Signature를 비밀키로 다시 계산해서 변조 여부 확인
4. 토큰에서 사용자 정보를 꺼낸다

### JWT 주의사항
만약에 토큰이 탈취 당했다면?
+ 유효기간을 설정한다
    + 서버 입장에서는 탈취 당했는지 아닌지 확인할 방법이 없음
    + 그래서 만약 탈취당했을 때 공격의 제한을 둬야함
+ 그리고 토큰 안에는 중요한 정보를 넣으면 안된다.
    + 변조가 어려울 뿐, 암호화 된 것은 아님
        + 서명은 변조 방지 역할임
        + 단순 인코딩 돼 있으므로 열람하기는 쉽다

### 로그인 플로우

1. 로그인 요청을 authToken으로 변환한다.
2. AuthenticationManager로 인증한다.
3. 인증 정보를 토대로 토큰을 발행한다.
4. 토큰을 클라이언트에 전달한다


# 관례적으로 Bearer 를 붙이는 이유
## Bearer
+ JWT 혹은 OAuth에 대한 토큰을 사용한다. (RFC 6750)
    + OAuth 2.0 개념
        + 비밀번호를 제공하지 않고 접근 위임을 위한 개방형 표준
        + 소셜 로그인 할때 사용하는 인증방식
+ Bearer의 뜻 - 소유자라는 뜻임, 이 토큰의 소유자에게 권한을 부여해줘 라는 의미
+ 생김새 구경
```json 
Authorization: Bearer {ACCESS_TOKEN}
```
